# MySQL Database Setup & Configuration Guide

This document provides detailed instructions for setting up the MySQL database for the **Smart Job Portal System**.

## 1. Installation Instructions (Windows)

### Option A: Automated (Using winget)
Run the following command in an Administrative PowerShell:
```powershell
winget install Oracle.MySQL
```
*Note: This will install the MySQL Community Server. You may need to follow the installer prompts to set the root password.*

### Option B: Manual
1.  Download the **MySQL Installer** from [dev.mysql.com](https://dev.mysql.com/downloads/installer/).
2.  Choose "Server only" or "Developer Default".
3.  Set the `root` password to `root` (to match our `application.properties`).
4.  Ensure the MySQL service is running.

---

## 2. Database Creation

Once MySQL is installed, open the **MySQL Command Line Client** or **MySQL Workbench** and run:

```sql
-- 1. Create the database
CREATE DATABASE jobportal_db;

-- 2. Verify creation
SHOW DATABASES;
```

---

## 3. Railway Cloud Setup (Recommended)

To use a cloud-based database on Railway.app:

1.  **Provision MySQL**: Create a new project on Railway and choose "Provision MySQL".
2.  **Environment Variables**: Railway will automatically provide variables like `MYSQLHOST`, `MYSQLUSER`, etc. The backend is configured to use these automatically.
3.  **Manual Database Creation**: Even on Railway, you must create the database schema once:
    *   Connect to your Railway MySQL instance using a local client (like MySQL Workbench).
    *   Run: `CREATE DATABASE jobportal_db;`
    *   *Note: If you use the Railway dashboard, you can also see the default database name (it might be `railway`), so you can either create `jobportal_db` or change the `MYSQLDATABASE` variable in Railway to `railway`.*

---

## 4. Spring Boot Configuration

The backend is already configured to connect to this database in `backend/src/main/resources/application.properties`:

```properties
spring.datasource.url=jdbc:mysql://localhost:3306/jobportal_db?useSSL=false&serverTimezone=UTC&allowPublicKeyRetrieval=true
spring.datasource.username=root
spring.datasource.password=root
spring.datasource.driver-class-name=com.mysql.cj.jdbc.Driver

# Hibernate will automatically create/update tables
spring.jpa.hibernate.ddl-auto=update
```

---

## 4. Table Structure & Attributes

The following tables will be automatically generated by Hibernate. Below is the detailed logical structure:

### Table: `users`
| Attribute | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | BIGINT | PRIMARY KEY, AUTO_INCREMENT | Unique identifier |
| `username` | VARCHAR(255) | UNIQUE, NOT NULL | User's login name |
| `password` | VARCHAR(255) | NOT NULL | BCrypt encoded password |
| `email` | VARCHAR(255) | UNIQUE, NOT NULL | Contact email |
| `role` | VARCHAR(50) | NOT NULL | `ADMIN`, `RECRUITER`, `JOB_SEEKER` |

### Table: `jobs`
| Attribute | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | BIGINT | PRIMARY KEY, AUTO_INCREMENT | Unique identifier |
| `title` | VARCHAR(255) | NOT NULL | Job title |
| `description` | TEXT | | Job details |
| `location` | VARCHAR(255) | | Job location |
| `recruiter_id` | BIGINT | FOREIGN KEY (`users.id`) | Links to the recruiter |
| `created_at` | DATETIME | NOT NULL | Auto-timestamp |

### Table: `applications`
| Attribute | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | BIGINT | PRIMARY KEY, AUTO_INCREMENT | Unique identifier |
| `applicant_id` | BIGINT | FOREIGN KEY (`users.id`) | Seeker who applied |
| `job_id` | BIGINT | FOREIGN KEY (`jobs.id`) | Job being applied for |
| `status` | VARCHAR(50) | DEFAULT 'APPLIED' | `APPLIED`, `SHORTLISTED`, etc. |
| `applied_at` | DATETIME | NOT NULL | Auto-timestamp |

### Table: `resumes`
| Attribute | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | BIGINT | PRIMARY KEY, AUTO_INCREMENT | Unique identifier |
| `user_id` | BIGINT | UNIQUE, FOREIGN KEY (`users.id`) | Owner of the resume |
| `file_name` | VARCHAR(255) | | Original filename |
| `file_type` | VARCHAR(100) | | e.g., `application/pdf` |
| `data` | LONGBLOB | | Binary content of the file |
| `uploaded_at` | DATETIME | NOT NULL | Auto-timestamp |

---

## 5. Relationships Summary
1.  **User ↔ Job**: One-to-Many (One recruiter can post many jobs).
2.  **User ↔ Application**: One-to-Many (One seeker can have many applications).
3.  **Job ↔ Application**: One-to-Many (One job can have many applicants).
4.  **User ↔ Resume**: One-to-One (One seeker has one primary resume).
